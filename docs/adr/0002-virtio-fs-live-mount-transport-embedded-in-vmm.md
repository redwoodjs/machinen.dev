# 0002 — virtio-fs live-mount transport, embedded in the VMM

#329 ported the host-side FUSE handlers for **Live mount** to Zig and closed most of the gap to docker, but a ~2× residual remained in the wire itself: per-request vsock RTT across tens of thousands of ops, a guest-side `/fuse-agent` byte-pump, and single-stream framing. #332 replaces that wire with a **virtio-fs** device — the same FUSE-derived protocol carried over a **VMM**-emulated virtqueue instead of over the **vsock bridge**.

The device is **embedded in the VMM**, not run as a sidecar and not delegated to upstream `virtiofsd`. `virtiofsd` was rejected for three reasons: it is Linux-only (it cannot run on the darwin/HVF host machinen also supports), it brings its own FUSE handlers and sandbox model — discarding the #329 Zig handlers and their `:ro` gate and lexical path-containment rules — and it speaks `vhost-user` to the VMM, which is arguably more protocol surface than parsing the virtqueue directly. A sidecar `mount-server` process talking to an in-VMM device over a pipe was also rejected: it keeps the lifecycle decoupling we already have for the fuse transport but reintroduces a host-process hop the embed approach removes entirely.

Embedding means the #329 opcode handlers are reused **verbatim**. They were extracted into a transport-agnostic `fuse` module (`packages/mount-server/src/fuse.zig`) that both consumers import: the standalone vsock `mount-server` and the in-VMM `virtiofs.zig` device. `dispatch()` takes one decoded FUSE request frame and returns one reply frame; each transport owns only its own framing — length-prefixed over a socket for vsock, scatter/gather across virtqueue descriptors for virtio-fs.

The virtio-fs path is **DAX-less for now**. The guest's reads and writes still copy through the virtqueue rather than mmap-ing host page cache directly. DAX is the gold-standard zero-copy mode on Linux/KVM but has no straightforward analogue under darwin/HVF's `hv_vm_map`, so it is deliberately left as a separate track — the framing win alone (no vsock RTT, no agent hop) is what #332 captures.

The VMM wires **one** virtio-fs device slot (slot 7 in `virt.dts`), so at most one `protocol: "virtiofs"` **Live mount** is served per **VM**; additional live mounts fall back to `protocol: "fuse"`, which is unbounded.

`protocol` **defaults to `"virtiofs"`** (only the first mount per VM claims the slot; any further unset mounts fall back to `"fuse"`). `scripts/bench/mount.ts` measured a steady ~1.8× wall-clock win over FUSE-over-vsock — short of the at-or-below-docker bar, but that bar is a Linux/KVM+DAX goal and a non-blocker for making virtio-fs the default. FUSE-over-vsock stays only as the explicit `protocol: "fuse"` fallback and is slated for **full removal** — the standalone `mount-server` process, the guest `/fuse-agent`, and the vsock mount channel all go once the registry/snapshot/restore paths no longer need to reason about two transports. Tracks #332; builds on #329; removal tracked in #338.

> **Superseded in part by [ADR 0003](0003-remove-fuse-over-vsock-transport.md) (#338).** FUSE-over-vsock and the `protocol` knob are gone; virtio-fs is the only live-mount transport. The single-slot constraint was lifted at the same time — the VMM now wires five virtio-fs slots.
